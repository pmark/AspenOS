Aspen Console - HELP
     INSTALLING A NEW APPLICATION

 
      

Required Files

In order to install a new application, Aspen needs a JAR (Java ARchive) file to be uploaded to the server. The files contained within the JAR need to appear in the directory structure described below.

/bin
/config
/data
/doc
/lib
/src (optional)
/sql
/templates
/xml

 

A script in an application's bin/ directory called dojar.sh has been provided in order to simplify JAR creation. Once you have changed the JARNAME and CLASSDIR variables in the dojar.sh script, running it will put all of the application's files in a JAR file in the lib/ directory.

See below for a description of each directory's contents.

  Destination       Description
  /bin   helpful scripts
  Contains scripts that simplify tedious tasks. The script dojar.sh is helpful to have with an installed application since much of the application data is taken from the app's JAR file at runtime. So a change to a properties or template file requires that the JAR be rebuilt in order for the app to take advantage of the change.
   
  /config   set up data
  Configuration settings for an application can be stored in properties files. Aspen provides a Properties Loader API that makes it very easy for your application to retrieve and reload these settings.
   
  /data   flat data files
  Not all application data has to be stored in a database. You can upload flat data files that your application can easily access with the Data File Loader API.
   
  /lib   Java class files, scripts, etc.
 
This directory must contain at least one JAR file with all of the Java class files used by your new application. Make sure the package structure conforms to the packaging guidelines.
   
  /src   (optional) source code for this application
  If you would like to have the source code for an application distributed along with the rest of the package, put it in the /src directory.
   
  /sql   files with SQL statements
  If you specify that Aspen should create a new database for your application, you will need to create tables, indices, stored procedures, and records required by your application. Aspen executes all valid SQL statements contained in each file in this directory at installation time.
   
  /templates   HTML, XML, and email variable templates
 

Templates are used for defining a large amount of nonchanging text along with tags that an application swaps with variable data. A template can be any text file that an application can load and perform tag substitutions on. Most templates contain HTML, XML or the body of an email message.

Any static HTML pages used by your application will easily and efficiently be served by Aspen. Static HTML filenames must end with the .html extension.
   
  /xml   XML docs used primarily for app installation
  Several XML docs are required for the operation of an Aspen application.
   
   
   

Packaging Guidelines

It is imperative that all applications use a common naming convention, or class name conflicts will occur. Using a reversed registered domain name for your package name is a good idea because registered domain names are guaranteed to be unique. For example, applications written by InternalRecords.com would be packaged like this:

com.internalrecords.aspenos.<app subpackage>


However, not every developer has access to a registered domain name. For these application developers, a unique AspenOS.com user ID should be used so that packages can be uniquely referred to as

com.aspenos.app.<user ID>.<app subpackage>

Following this simple protocol will ensure a working environment for everybody, while apps that fail to do so will be rejected by Aspen or will not work properly.
